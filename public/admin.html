Integración con Meta Pixel

Configura el Pixel de Facebook/Instagram y la verificación de dominio. Esta página genera el archivo de configuración; tú lo descargas, lo subes al repo y vuelves a desplegar.

🔒 Esta página no debe enlazarse públicamente. No guarda secretos y no puede cambiar producción por sí sola: solo genera el archivo meta-config.js.

1 Configuración del Pixel

Toma estos datos de Meta Events Manager → tu Pixel → Configuración.

2 Verificación de dominio

Meta pide verificar el dominio. El método recomendado es subir el archivo HTML que te da Meta, porque no depende de JavaScript.

  • A) Archivo HTML Recomendado
    En Meta: Configuración del negocio → Seguridad de la marca → Dominios → tu dominio → Subir archivo HTML. Descarga el archivo xxxxxxxx.html y colócalo en public/ (queda servido en la raíz del sitio). Commit → push → Deploy → pulsa “Verificar” en Meta.
  • B) Etiqueta <meta> estática
    Pega esta línea dentro del <head> de public/index.html: 
    <meta name="facebook-domain-verification" content="" />
    Escribe el código arriba para rellenar esta etiqueta. Súbela tú porque el rastreador de Meta no ejecuta JavaScript de forma fiable.
  • C) Registro DNS TXT
    Si tienes acceso al DNS del dominio, añade el TXT que indica Meta. Útil si no quieres tocar archivos.

3 Conversions API (eventos de servidor)

Envía los eventos también desde el servidor (además del Pixel) con deduplicación por event_id. Mejora la atribución cuando el navegador bloquea cookies/JS. El token es secreto y va en Dokploy, nunca en el repo.

⚙️ Para que funcione debes desplegar con Compose (incluye el servicio capi) y definir estas variables de entorno en Dokploy:
  • CAPI_ENABLED = true
  • CAPI_ACCESS_TOKEN = token de la Conversions API (Events Manager → Configuración → Conversions API → Generar token de acceso) secreto
  • CAPI_PIXEL_ID = el mismo Pixel ID de arriba
  • CAPI_TEST_EVENT_CODE = (opcional) código de “Probar eventos” para ver los eventos de servidor en tiempo real
  • CAPI_ALLOWED_ORIGIN = (opcional) https://tudominio.com para rechazar otros orígenes

El campo “Pixel ID” de arriba se escribe en el archivo para el navegador. El backend usa su propia variable CAPI_PIXEL_ID (mismo valor) más el token, que nunca se guarda en archivos del sitio. Comprueba el estado del backend en /api/capi/health.

      Notas avanzadas (CAPI, noscript, seguridad del panel)
      

        
Conversions API (CAPI): es de servidor y no se puede hacer desde un sitio estático sin un backend (el token nunca debe ir en archivos del navegador). Queda fuera de alcance; opciones futuras: una función serverless en tu VPS o el CAPI Gateway de Meta.

        
Test Event Code: aplica a CAPI/servidor; el Pixel del navegador lo ignora, por eso no se escribe en el archivo. Para probar usa Events Manager → Probar eventos.

        
<noscript>: se omite a propósito. Si JS está desactivado el Pixel no puede ejecutarse de ninguna forma; el tráfico sin JS es mínimo.

        
Proteger este panel: ya lleva noindex y está en robots.txt. Para pedir usuario/contraseña, descomenta el bloque de HTTP Basic Auth en nginx.conf (instrucciones incluidas).
